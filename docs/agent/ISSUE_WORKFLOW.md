# Issue 开发工作流

每个 Issue 进入编码前，需完成以下讨论与确认。适用于整个项目周期，不限于特定 Issue。

---

## Step 0：上下文同步（每个新 Session）

按 AGENTS.md 规定的上下文同步协议执行：

```
请先阅读：
README.md / ARCHITECTURE.md / AGENTS.md / LEARNING_MODE.md
docs/issues/ 目录下当前正在进行的 Issue

阅读完成后总结：
1. 当前项目目标
2. 当前架构
3. 当前完成进度（已完成的 Issue 及关键产出）
4. 当前正在处理的 Issue
5. 你的开发约束

确认无误后等待我的任务。
```

---

## Step 1：Issue 理解

在写任何代码前，先回答：

1. 这个 Issue 在整个系统中的位置是什么？（属于哪个模块？被谁调用？调用谁？）
2. 它的职责边界是什么？（做什么 / 不做什么）
3. 它与哪些已有模块有交互？
4. 完成标准是什么？（怎么判断做完了？）

---

## Step 2：设计方案（Level 2 决策点）

此阶段**只输出方案，不写代码**。

需包含：

1. 有哪些候选方案？
2. 推荐哪个方案？为什么？
3. 不推荐的方案为什么被排除？
4. 该方案牺牲了什么？（违反了什么原则？选择了哪种取舍？）
5. 该设计未来可能出现什么问题或限制？

涉及以下情况时必须暂停，等待确认：
- 新增 public 类 / interface
- 决定某个职责放在哪个模块
- 设计对外暴露的 API
- 是否引入抽象

---

## Step 3：等待确认

Agent 输出方案后，不继续执行，等待用户反馈：

- 方案通过 → 进入 Step 4
- 需要修改 → 返回 Step 2
- 范围需要调整 → 更新 Issue 描述

---

## Step 4：实现

AGENTS.md 权限约束生效：
- Level 1 操作直接执行
- Level 2 / Level 3 操作需先说明

实现过程中：
- 不修改当前 Issue 范围外的文件
- 不引入未经批准的依赖
- 不提前实现后续 Issue 的功能
- 每完成一个核心类时暂停，说明类职责、与哪些类交互、为什么这样写，再继续

---

## Step 5：Review（按 LEARNING_MODE.md 格式输出）

完成后固定输出：

```
【修改文件】
【为什么这样设计】
【这一阶段必须理解】（≤5 条）
【这一阶段可以暂时不用理解】
【验证方式】
【建议停止】
```

---

## Step 6：验证 & 记录

1. 运行验证命令，确保通过
2. 检查：职责是否正确？依赖方向是否正确？是否过度设计？
3. 提交 Git Commit（一个 Issue → 一次 Commit）
4. 更新 CHANGELOG.md
5. 更新 TODO.md（标记完成）

完成后立即停止，等待用户 Review。

---

## 文档节制原则

文档服务于开发。禁止为了完善文档而创建文档。

只有以下情况需要新增文档：
1. 新架构决策（ADR）
2. 新技术选择
3. 长期维护信息
4. 后续开发者必须知道的信息

判断标准：这个文档半年后还有价值吗？如果没有，不需要创建。
